# 需求澄清 Prompt 模板

## 使用场景

- 收到模糊需求（「做一个 XX 功能」）
- 改需求前确认影响范围
- 写 PRD 前的结构化澄清

---

## Prompt 正文 — 澄清问题清单

```
你是一名资深 [产品/技术] 负责人。用户提出以下需求描述，请输出澄清问题清单。

## 输出结构

### 1. 需求理解（你的复述，1 段）
用一句话复述你认为的需求是什么。标注不确定处 [?]

### 2. 澄清问题（按类别分组，每类 3–5 个）
- 目标与用户：谁用？解决什么痛点？成功标准？
- 范围与边界：做什么 / 不做什么？
- 流程与交互：主流程步骤？异常分支？
- 数据与集成：输入输出？依赖哪些系统/API？
- 非功能：性能、安全、合规、上线时间？
- 优先级：MVP 最小范围是什么？

### 3. 风险假设（可能踩坑的 3 点）

### 4. 建议的 MVP 切分（如适用）

## 规则
- 问题要具体、可回答，避免空泛
- 不要假设未给出的业务规则
- 中文

## 需求描述
【粘贴区】
```

---

## Prompt 正文 — PRD 骨架（澄清完成后）

```
根据以下已澄清的需求信息，生成 PRD 骨架（不是完整 PRD）。

结构：
1. 背景与目标
2. 用户故事（3–5 条，As a... I want... So that...）
3. 功能范围（In scope / Out of scope）
4. 主流程（步骤列表）
5. 验收标准（可测试的 Given-When-Then 3–5 条）
6. 依赖与风险
7. 待决事项

已澄清信息：
【粘贴澄清结果、会议纪要、聊天记录摘要】
```

---

## Prompt 正文 — 技术方案预审（研发用）

```
以下是产品 PRD 摘要。作为 tech lead，输出：
1. 技术可行性：高/中/低 + 理由
2. 预估工作量：前端/后端/测试 各 X 人天（区间）
3. 架构影响：新组件 / 改现有 / 第三方
4. 必须在开工前确认的技术问题（清单）
5. 建议的 task 拆分（5–10 个）

PRD 摘要：
【粘贴区】
```

---

## 工作流

```
模糊需求 → 澄清问题 Prompt → 开会/异步回答 → PRD 骨架 Prompt → 人工补全 → 开发
```

---

## 效果度量

- 目标：减少 1 轮需求往返（baseline 记录澄清轮次）
- 追踪：每个需求记「澄清轮次」和「从需求到开工天数」
